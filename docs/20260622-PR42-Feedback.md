# 20260622-PR42-Feedback

This document consolidates the findings discovered during the deep review pass and includes recommended fixes for each issue.

| # | Priority | Finding | Details / Risk | Recommended Fix |
|---|---:|---|---|---|
| 1 | P1 | FinServ multi-region results are not region-correct | FinServ runs only on `RegionIndex == 0`, creates clients in the Lambda/default region, then labels findings for multiple target regions. This can report controls as assessed in regions that were never queried. | Either run FinServ per region like the other services, or make FinServ explicitly account/global-scoped and remove per-region stamping. Only attach a region when the API call was made in that region. |
| 2 | P1 | FinServ can exceed Step Functions payload limits | FinServ returns all findings in the Lambda response body. Large accounts can exceed the 256 KiB Step Functions state payload limit even if CSV upload succeeded. | Return only metadata such as `s3_url`, `findings_count`, and `status`; keep full findings in S3. Add a regression test with a large synthetic finding set. |
| 3 | P1 | Lambda `statusCode: 500` does not fail Step Functions | Several handlers catch fatal exceptions and return a 500 payload. Optimized Lambda integration treats the task as successful unless the Lambda raises or ASL checks the payload. | For fatal errors, raise exceptions. Alternatively add ASL `Choice` states after each Lambda invocation to fail when `Payload.statusCode >= 400`. |
| 4 | P1 | Parallel assessment branch failures are hidden | Branch `Catch` handlers replace failures with empty findings and continue, so the final report can look complete while an entire service failed. | Preserve incomplete-service markers in S3/report output and fail the execution for required services unless explicitly configured to tolerate partial results. |
| 5 | P1 | Report-generation failure can still look successful | The consolidated report Lambda also returns `statusCode: 500` instead of failing the state. | Make report generation exceptions fail the Lambda invocation. Add an integration test that simulates S3/report write failure and expects Step Functions failure. |
| 6 | P1 | Multi-account can succeed with no current report data | If no findings are found, `consolidate_html_reports.py` prints a message and exits 0, so CodeBuild reports success without a consolidated report. | Treat zero input CSVs as an error unless an explicit empty-scan-allowed flag is set. Exit non-zero and include account IDs missing expected files. |
| 7 | P1 | Stale historical CSV/HTML files can enter current reports | Buildspec copies/syncs every `*.csv` and `*.html`, not only the current execution artifacts. Old findings can be presented as current evidence. | Scope all copy/sync/consolidation to the current execution ID. Prefer S3 prefixes like `executions/{execution_id}/...` over bucket-root filenames. |
| 8 | P1 | Cleanup failure does not stop stale-data reuse | Cleanup returns a 500 payload instead of raising, and the state ignores it. Old files may remain and be copied later. | Make cleanup failure fatal or move to execution-id-scoped output so cleanup is no longer correctness-critical. |
| 9 | P1 | Multi-account SAM template is stale versus single-account | Buildspec uses `template-multi-account.yaml`, but it has older timeout/permission behavior than `template.yaml`. | Either eliminate the duplicate template or add parity tests that compare required functions, timeouts, environment variables, and IAM actions across both templates. |
| 10 | P1 | Multi-account FinServ IAM is incomplete | Missing actions include newer FinServ dependencies such as `bedrock:ListTagsForResource`, `bedrock:ListAutomatedReasoningPolicies`, and `scheduler:ListSchedules`. | Update multi-account IAM to match the actual FinServ call graph and include it in IAM coverage tests. |
| 11 | P1 | Multi-account template contains invalid/stale IAM actions | Example: `bedrock:ListModelInvocations` appears to be invalid; stale ECR action drift also exists. | Validate every IAM action with IAM Access Analyzer/cfn-lint where possible and botocore service models. Remove nonexistent actions. |
| 12 | P1 | IAM coverage tests exclude deployed multi-account template | `tests/test_iam_coverage.py` covers single-account only, missing the path used by multi-account builds. | Parameterize the IAM coverage test over both SAM templates and the deployment/member-role templates. |
| 13 | P2 | Bedrock `ListAgents` response shape is wrong | Code reads `agents`; current API returns `agentSummaries`. Agent checks can falsely report no agents. | Replace all `agents` reads with `agentSummaries`; update tests to use botocore Stubber or real modeled fixtures. |
| 14 | P2 | Bedrock `GetAgent` response shape is wrong | Code reads top-level `agentResourceRoleArn`; current API nests it under `agent`. | Read `get_agent(...).get("agent", {}).get("agentResourceRoleArn")`. Add a test matching botocore shape. |
| 15 | P2 | Bedrock footprint detection can return false N/A | The footprint probe uses the same wrong `agents` key, so agent-only regions can be treated as having no Bedrock resources. | Fix the probe shape and add tests for regions with only agents, only KBs, only prompts, and only flows. |
| 16 | P2 | Bedrock Prompt Management uses wrong parameter | `GetPrompt` requires `promptIdentifier`; code passes `promptId`. Variant checks can silently fail. | Use prompt summary `id` or `arn` as `promptIdentifier`. Treat detail-call failures as assessable warnings, not silent skips. |
| 17 | P2 | Bedrock prompt active/draft counts are inaccurate | `ListPrompts` summaries do not expose `status`, but code counts `ACTIVE`/`DRAFT`. | Remove active/draft counts or fetch versions/details only where the API supports the desired fields. |
| 18 | P2 | Bedrock Knowledge Base IAM gaps | Code calls KB/data-source APIs, but Lambda IAM does not fully grant those actions in all templates. | Add `bedrock:ListKnowledgeBases`, `GetKnowledgeBase`, `ListDataSources`, and related required actions consistently across deployment paths. |
| 19 | P2 | Bedrock Lambda relies on managed boto3 for newer APIs | `requirements.txt` only has `pydantic`; newer Bedrock APIs may not exist in the Lambda runtime SDK. | Vendor/pin `boto3` and `botocore` in Bedrock requirements, aligned with the minimum API model used by the code. |
| 20 | P2 | AgentCore `ListGateways` response key is wrong | Current API returns `items`; code reads `gateways`, so gateway checks often report no gateways. | Read `items`; add modeled tests for non-empty gateway lists. |
| 21 | P2 | AgentCore `GetGateway` parameter is wrong | Current API requires `gatewayIdentifier`; code passes `gatewayId`. | Call `get_gateway(gatewayIdentifier=gateway_id_or_arn)`. |
| 22 | P2 | AgentCore memory encryption reads wrong response level | `GetMemory` returns `{"memory": {...}}`; code reads top-level `encryptionKeyArn`. | Read `memory_details.get("memory", {}).get("encryptionKeyArn")`. |
| 23 | P2 | AgentCore browser recording check models wrong resource | Browser recording is exposed through `ListBrowsers`/`GetBrowser.recording`, not runtime `storageConfig`. | Rework AC-06 to list browsers, fetch each browser, and inspect `recording`. |
| 24 | P2 | AgentCore browser IAM is missing | Templates do not grant browser list/get actions. | Add `bedrock-agentcore:ListBrowsers` and `bedrock-agentcore:GetBrowser` if AC-06 remains in scope. |
| 25 | P2 | AgentCore VPC subnet validation reads wrong shape | Runtime subnets are under `networkConfiguration.networkModeConfig.subnets`; code reads `subnetIds`. | Update parsing and test both `PUBLIC` and `VPC` runtime shapes from botocore. |
| 26 | P2 | AgentCore observability checks nonexistent fields | Code checks `loggingConfig` and `tracingConfig` on `GetAgentRuntime`, which current model does not expose. | Replace with supported observability signals, likely CloudWatch/X-Ray delivery configuration or documented AgentCore observability APIs. |
| 27 | P2 | AgentCore resource-policy APIs are wrong | Current operation is `GetResourcePolicy(resourceArn)`, not runtime/gateway-specific methods. | Use resource ARNs from runtime/gateway responses and call `get_resource_policy(resourceArn=...)`. |
| 28 | P2 | AgentCore resource-policy IAM actions are wrong | Templates grant non-modeled resource-policy actions. | Replace with the correct `bedrock-agentcore:GetResourcePolicy` action if supported by IAM docs/policy simulator. |
| 29 | P2 | Multi-account execution-start failures are warnings | If `start-execution` fails, build continues. | Mark the account failed and exit non-zero unless an explicit partial-scan mode is enabled. |
| 30 | P2 | Missing execution ARN is only a warning | Post-build can continue when an account has no saved execution ARN. | Treat missing execution ARN as failure and skip copying artifacts for that account. |
| 31 | P2 | Multi-account poll timeout is only a warning | Timeout waiting for Step Functions does not set `sf_failed=true`. | Set failure state on timeout and do not copy/report results for incomplete executions. |
| 32 | P2 | `MultiAccountListOverride` has inconsistent management-account behavior | Management account is always run, but post-build only processes override accounts. | Define expected semantics. Either always include management in the effective account list, or only run it when present/when override is empty. |
| 33 | P2 | `ConcurrentAccountScans` does not implement account concurrency | It selects CodeBuild size and exports `PARALLEL_ACCOUNTS`, but account loop is sequential. | Implement bounded parallelism with `xargs -P`, background jobs with wait/error aggregation, or rename the parameter to reflect compute size only. |
| 34 | P2 | Report HTML is not escaped | CSV fields are interpolated into HTML attributes/cells and links, creating injection risk. | Use `html.escape` for text/attributes and validate `Reference` URLs against allowed schemes before rendering links. |
| 35 | P2 | Tests encode incorrect AWS API shapes | MagicMock fixtures use wrong keys like `agents` and `gateways`, masking runtime bugs. | Replace these fixtures with botocore Stubber responses or a small shape-validation helper using botocore models. |
| 36 | P2 | Report Lambda ignores branch outputs | Map/Parallel outputs are discarded with `ResultPath: null`; report reads only S3 files. | Persist per-service completion metadata to S3, or keep compact status outputs and include incomplete/failed branch state in the report. |
| 37 | P3 | Deployment wrappers do not expose `MaxRegionConcurrency` | Inner SAM supports it, but deployment templates expose only `TargetRegions`. | Add wrapper parameters for `MaxRegionConcurrency` and pass them through `sam deploy --parameter-overrides`. |
| 38 | P3 | README overpromises no added multi-region time | Real duration depends on `MaxRegionConcurrency`, Lambda duration, throttling, and service availability. | Change docs to "parallel up to configured concurrency" and describe default/maximum behavior. |
| 39 | P3 | README says `ConcurrentAccountScans` adjusts concurrency | Current implementation does not parallelize account scans. | Fix implementation or update docs to say it only changes CodeBuild compute size. |
| 40 | P3 | Troubleshooting docs claim parallel account scans | Docs show parallel account counts that are not honored by buildspec. | Align troubleshooting with actual behavior after deciding whether to implement real parallelism. |
| 41 | P3 | FinServ FS-01 docs overstate detection | Docs claim CloudFront WAF/body-size coverage, while code checks regional WAF scope. | Either add CloudFront-scope WAF/body-size checks or narrow the docs to regional WAF coverage. |
| 42 | P3 | FinServ docs overstate evidence depth | Several controls claim deep `Get*` evidence while code lists summaries or emits advisory findings. | Audit every FS doc row against the exact API calls. Mark advisory/manual controls clearly. |
| 43 | P3 | FinServ permissions-gap wording does not match implementation | Docs say missing permissions are reported as `COULD NOT ASSESS`, but report statuses are Passed/Failed/N/A. | Add a distinct status if desired, or update docs to match current status taxonomy. |
| 44 | P3 | FinServ report wording implies per-region evidence | Because much FinServ inventory is collected once, per-region UI labels can imply stronger regional evidence than exists. | Add explicit scope metadata per check: `regional`, `global`, `advisory`, or `stamped`. Render that in report details. |
| 45 | P3 | FinServ timeout differs across templates | Single-account uses 900s; multi-account uses 600s. | Standardize timeout or document why multi-account intentionally has less time. |
| 46 | P3 | Sample reports include invalid/stale permission examples | Sample output includes invalid action names, which can mislead users and reviewers. | Regenerate sample reports after IAM cleanup and add a test preventing known invalid action strings. |
| 47 | P3 | Multi-account consolidator has no execution-id scoping | It reads all matching CSVs in each account directory. | Pass current execution IDs into the consolidator and filter filenames or use execution-specific directories. |
| 48 | P3 | Single-account result sync lacks execution-id scoping | It syncs all CSV/HTML files to the report bucket account prefix. | Sync only keys matching the current execution ID, or write outputs under `executions/{execution_id}/`. |
